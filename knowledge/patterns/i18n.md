# Pattern: Internacionalização (i18n)

Três contextos com abordagens completamente diferentes.

## API — i18n de conteúdo

Conteúdo multilingual armazenado em tabelas de tradução separadas.
Locales suportados: `pt-BR`, `en`, `es`.

Existem duas estratégias — ver critério em `decisions/002-i18n-strategy.md`:

### i18n frouxa

Entidade aceita locales parciais — pode ser criada com apenas 1 tradução.
Resolução de locale faz fallback para o locale disponível mais próximo.

```ts
// criação com locale único (válido)
POST /resources
{
  "translations": [
    { "locale": "pt-BR", "name": "Nome do recurso" }
  ]
}
// GET /resources/:id?locale=en → retorna pt-BR como fallback se en não existir
```

### i18n estrita

Entidade exige **todos os locales** ao criar ou atualizar. Validação no service.

```ts
// INVÁLIDO — rejeita com AppError
{ "translations": [{ "locale": "pt-BR", "name": "Nome" }] }  // faltam en e es

// VÁLIDO
{ "translations": [
  { "locale": "pt-BR", "name": "Nome" },
  { "locale": "en", "name": "Name" },
  { "locale": "es", "name": "Nombre" }
] }
```

### Resolução de locale (Accept-Language)

A função `resolveLocale` em `src/lib/locale/resolve-locale.ts` parseia o header `Accept-Language`
e retorna o locale mais adequado:

```ts
resolveLocale('en-US,en;q=0.9,pt-BR;q=0.8') // → 'en' (match por base 'en')
resolveLocale('fr,de')                         // → 'pt-BR' (fallback — nenhum suportado)
resolveLocale(undefined)                       // → 'pt-BR' (fallback padrão)
```

Ordem de prioridade: match exato → match por língua base (ex: en-US → en) → fallback `pt-BR`.

### E-mails (API)

```
src/lib/mail/
├── content/
│   ├── pt-BR/    ← textos em português
│   ├── en/       ← textos em inglês
│   └── es/       ← textos em espanhol
└── emails/       ← templates React Email
```

## Admin — UI monolíngue pt-BR

O painel admin **nunca** é traduzido. Todos os textos de interface são pt-BR fixos.

- Sem `i18next`, sem `next-intl`, sem qualquer lib de i18n
- Labels, mensagens de erro, botões: sempre pt-BR
- Conteúdo multilingual das entidades é editado em pt-BR no admin (locale selecionado pelo usuário)

## Web — UI multilingual com next-intl

> **Quando usar:** apps/web que precisa exibir a interface em múltiplos idiomas para o visitante
> (pt-BR, en, es). Se o web for monolíngue ou não tiver público internacional, `next-intl`
> é desnecessário — use strings hardcoded como no admin.

Toda a interface do site é traduzida em 3 idiomas.

### Configuração

```ts
// locales: pt-BR (padrão), en, es
// estratégia: as-needed — pt-BR em /, en em /en/, es em /es/
```

### Uso em Server Components

```ts
import { getTranslations, getLocale } from 'next-intl/server'

export default async function Page() {
  const t = await getTranslations('Home')
  const locale = await getLocale()
  return <h1>{t('title')}</h1>
}
```

### Uso em Client Components

```tsx
'use client'
import { useTranslations, useLocale } from 'next-intl'

export function Header() {
  const t = useTranslations('Nav')
  const locale = useLocale()
  return <nav>{t('tours')}</nav>
}
```

### LocaleSwitcher

Componente no Header para troca de idioma. Exibe bandeiras (PT/EN/ES).
Ao trocar: redireciona para o mesmo path com o novo locale prefixado.
